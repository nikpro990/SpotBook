# SpotBook — бронирование мест

[![CI](https://github.com/ВАШ_ЛОГИН/spotbook/actions/workflows/ci.yml/badge.svg)](https://github.com/ВАШ_ЛОГИН/spotbook/actions/workflows/ci.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)

**SpotBook* — это Fullstack-приложение для управления бронированием рабочих мест, кресел или конференц-залов в режиме реального времени. Проект спроектирован с упором на строгую типизацию, отказоустойчивую бэкенд-архитектуру, контейнеризацию и защиту от бизнес-ошибок.

## Стек
 
| Слой | Технологии |
|---|---|
| Фронтенд | React 18, TypeScript, Vite |
| Бэкенд | Node.js, Express, TypeScript |
| База данных | PostgreSQL 16, миграции через node-pg-migrate |
| Контейнеризация | Docker, Docker Compose |

## Запуск одной командой

Для работы приложения требуются установленные **Docker** и **Docker Compose**.

1. Создайте файл `.env` в корневой папке проекта и заполните переменные окружения:

```env
# --- Настройки базы данных PostgreSQL ---
POSTGRES_USER=spotbook
POSTGRES_PASSWORD=your_secure_password_here
POSTGRES_DB=spotbook

# --- Настройки Frontend ---
FRONTEND_ORIGIN=http://localhost:5173

# --- Настройки SMS-шлюза (Опционально) ---
SMS_GATEWAY_URL=https://sms-provider.com
SMS_GATEWAY_API_KEY=your_api_key_from_provider_dashboard
```

2. Запустите сборку и старт всех контейнеров:
```bash
docker compose up --build -d
```


Затем:

```bash
docker compose up --build
```

Что поднимется:

| Сервис    | Контейнер   | Порт  | Назначение                                          |
|-----------|-------------|-------|-------------------------------------------------------|
| db        | postgres:16 | —     | Изолированная БД (порт наружу не пробрасывается)       |
| backend   | node:20     | 4000  | REST API на TypeScript, миграции применяются автоматически при старте |
| frontend  | node:20     | 5173  | Мобильный веб-интерфейс (React + TypeScript)           |

Приложение: **http://localhost:5173**
API напрямую: **http://localhost:4000/api/health**

При первом запуске backend-контейнер сам применяет все миграции из
`backend/migrations/` (схема + демо-данные: 3 зоны с картой мест) — это
происходит в `CMD` контейнера, до старта сервера (см. `backend/Dockerfile`).

## Безопасность

- **Валидация всех входных данных через `zod`** — номер телефона, даты, часы,
  количество мест и их id проверяются на сервере перед любой обработкой.
- **Секреты и OTP-коды не хранятся в открытом виде** — в БД только SHA-256
  хэш кода подтверждения и хэш сессионного токена.
- **Сессионный токен** — 256 бит случайности, срок действия 30 дней,
  timing-safe сравнение при входе.
- **Лимит попыток ввода OTP-кода** (5) и антиспам-пауза между запросами
  кода (60 сек на номер).
- **Rate limiting на уровне API** — общий лимит на `/api`, более строгий на
  `/api/auth`.
- **Проверка принадлежности мест локации** — нельзя забронировать места
  одного зала под видом другого через прямой вызов API.
- **Проверка бизнес-инвариантов**: нельзя бронировать прошедшую дату,
  ограничена максимальная длительность и число мест в одной брони.
- **HTTP security-заголовки** через `helmet`, CORS ограничен конкретным
  доменом фронтенда, порт БД не пробрасывается наружу, секреты вынесены в `.env`.
- **SQL-инъекции** — везде параметризованные запросы, конкатенация строк в
  SQL нигде не используется.
- **TypeScript strict-режим** на бэкенде и фронтенде — весь проект проходит
  `tsc --noEmit` без единой ошибки, что отсекает целый класс ошибок ещё на
  этапе сборки (несовпадение форм данных между слоями, пропущенные
  приведения типов и т.п.).

## Индексы и производительность БД

Миграция `1721380000000_init-schema.js` создаёт индексы под реальные
паттерны запросов приложения:
- `seats(location_id)` — карта мест локации (самый частый запрос);
- `bookings(user_id)` — история бронирований пользователя;
- `bookings(booking_date, location_id)`;
- `booking_slots(slot_date, slot_hour)` — проверка занятости при построении карты.

## Что стоит добавить для боевого продакшна

- HTTPS/TLS-терминация перед фронтендом и бэкендом (nginx/traefik/Caddy).
- Отдельная роль PostgreSQL с правами только на нужные таблицы (сейчас
  `spotbook` — владелец БД).
- Логирование и мониторинг подозрительной активности.
- CAPTCHA на запрос OTP-кода при появлении массового спама.
- CI-пайплайн: `npm run typecheck` + `npm test` + `npm run build` на каждый PR.
- Регулярные бэкапы БД (`pg_dump` в отдельное хранилище) и/или managed-БД
  с репликацией.
- Панель администратора для управления залами/местами (сейчас данные
  задаются миграцией-сидом).
